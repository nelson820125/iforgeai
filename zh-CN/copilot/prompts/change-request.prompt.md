对当前进行中的迭代处理一项需求变更。
按以下步骤依次执行。**不要跳过影响评估** — 先收集变更详情，再评估范围。

---

## 步骤 1：记录变更

询问用户：

> **CR1.** 什么在变化？描述改动本身——不是完整功能，只是新增、删除或修改的部分。
>
> **CR2.** 为什么需要这个变更？（业务原因、发现的约束或干系人决定）
>
> **CR3.** 这个变更纳入当前迭代，还是推迟到下一个 Sprint？
> 可选：`纳入` / `推迟`

若选择`推迟` → 将变更写入 `.ai/context/backlog.md`（不存在则创建，已存在则追加），到此结束。

---

## 步骤 2：影响评估

评估前先读取以下文档：

- `.ai/temp/requirement.md`
- `.ai/temp/architect.md`
- `.ai/temp/db-design.md`
- `.ai/temp/ui-design.md`
- `.ai/temp/wbs.md`
- `.ai/temp/api-contract.md`
- `.ai/temp/plan.md`

对每份文档判断：**此变更是否需要更新该产出物？**

输出影响评估表：

| 产出物 | 受影响？ | 变更内容 |
|---|---|---|
| `requirement.md` | 是 / 否 | （描述改动） |
| `architect.md` | 是 / 否 | ... |
| `db-design.md` | 是 / 否 | ... |
| `ui-design.md` | 是 / 否 | ... |
| `wbs.md` | 是 / 否 | ... |
| `api-contract.md` | 是 / 否 | ... |
| `plan.md` | 是 / 否 | ... |

---

## 步骤 3：重新执行方案

根据影响评估表，说明需要重新执行哪些阶段及其顺序：

```
需重新执行：P{N} → P{N+1} → ...
不受影响（跳过）：P{X}、P{Y}
```

执行前请用户确认：

> 此变更需要重新执行：**[列出阶段]**
> 确认继续？（是 / 调整 / 取消）

---

## 步骤 4：执行

确认后，指导用户在 `@digital-team` 中点击对应的移交按钮，依次重新执行每个受影响的阶段。

将变更日志写入 `.ai/context/change-log.md`（不存在则创建，已存在则追加）：

```markdown
## CR-{日期} · {一行变更摘要}

**原因：** {CR2 答案}
**重新执行阶段：** {列表}
**更新的产出物：** {列表}
```
